# Fleet coordination race matrix (flow-8by2.1 spike deliverable)

**Purpose.** This is the spike deliverable for epic flow-8by2 (fleet coordination authority), gating children 2-5. It inventories every coordination read and write across flow's run fleet, maps the liveness/work-state sources (L lease, M launch markers, R PR/branch refs, B bead status, J jobs-dir scan), enumerates every read-classify-then-mutate TOCTOU window the per-subsystem mappers found (with the flock-held flag and the bug class each window caused or that closed it), folds in the Agent-Teams build-vs-adopt comparison, and returns a GO/NO-GO verdict plus per-child sizing. The thesis under test: liveness is currently inferred from a five-source eventually-consistent join across six independent join sites, the architecture generates a coordination-bug class faster than point patches retire it (six coordination fixes landed on 2026-06-10 alone), and a single `.flow/fleet/` flock-serialized registration+heartbeat+deregistration ledger would collapse the three liveness sources (lease + launch markers + jobs scan) into one authority while gh/bd stay as work-state reads. Conclusion up front: the TOCTOU surface is broad and recurring enough to justify one unifying ledger, and build beats adopt on the constraints that matter. Verdict: **go**, staged, with two non-negotiables (a heartbeat-staleness fallback, and a property-equivalence proof before any retirement).

**Status update (flow-8by2.5, child-5 landed).** Source **M** (launch_ledger) is retired: `launched_pending` is now derived from the fleet ledger (`fleet_keys - live_keys`, `_evolve_common.fleet_live_keys` minus the lease-only set), gated on the executable property-equivalence proof this doc's verdict required (`tests/test_launch_window_equivalence.py`). Source **J** (jobs-dir scan) was NOT retired — demoted to a permanent pre-first-register crash fallback, per the session-cleanup regression this doc flags in §4 and §6.

## 1. Sources, and the consolidation note

Five sources feed the fleet's liveness/work-state picture. Three are liveness (who is running), two are work-state (what the work is doing). The ledger targets the three liveness sources only.

| Src | What it is | Liveness or work-state | Authoritative reader primitive |
|---|---|---|---|
| **L** | per-ticket run lease (`run.lock` JSON under `run.lock.lock` flock) | liveness | `lease.classify()` (lock-free read), `lease.classify_then()` / `takeover_clear()` (under flock) |
| **M** | launch-ledger TTL markers (`.flow/launch-ledger/<key>`, 1800s) — **RETIRED flow-8by2.5** | liveness (launch->init blind window only) | ~~`launch_ledger.live_keys()`~~ -> `fleet_keys - live_keys` (`_evolve_common.fleet_live_keys` minus the lease-only set) |
| **R** | open-PR head refs + git branch refs | work-state (PR/branch lifecycle), liveness-adjacent | `_evolve_common.gather_refs()` (`gh pr list` + `git for-each-ref`) |
| **B** | bead status/labels (Dolt truth via `bd`) | work-state (open/in_progress/closed/deferred/blocked) | `bd ready` / `bd list` / `bd show` |
| **J** | `~/.claude/jobs/*/state.json` bg-session scan | liveness (session presence; schema-less, no `pid`) | `evolve_session_cleanup._enumerate_jobs()` |

**Consolidation note (read before Section 2).** The seven per-subsystem maps carry ~42 TOCTOU entries, but many are the *same race observed from different files*. The select-time four-source union shows up in the launch_ledger map, the evolve_select map, and the _evolve_common map; the reap classify->merge gap shows up in the lease, evolve_reap, version_remerge, and _evolve_common maps. The distinct-race count is ~13. Section 2 is organized by source x operation and consolidates to distinct races, but every cell lists **all** the file:line citations the mappers gave that race, so a reviewer can trace all ~42 mapper entries back to a row. Where a row consolidates N sighting sites, it says so.

## 3. CROSS-CUTTING FINDINGS

**Six independent join sites.** The five-source liveness/work-state picture is reassembled from scratch at six places, each with its own union logic and its own subset of sources:

1. `evolve_select.select` — B + R + L + M (the full four-channel union + hot read)
2. `evolve_drain.cli_main` — R + L + M + B (re-reads L via liveness_map, reconciles M)
3. `queue_select.select` — R + L + M + B (day-job cap scoping)
4. `queue_drain.cli_main` — L + R + B (merged-PR reap, evolve-scoped marker subtract)
5. `evolve_reap.reap` — R + B + L + R(CI) (orphan-merge classify + main-CI gate)
6. `evolve_session_cleanup.classify` — J + L + B (bg-session tombstone)

No two of them compute liveness the same way, and three of them (1, 2, 5) read L twice within one drain turn through different code paths.

**The same liveness fact computed inconsistently (the direct motivation for one authority).** The cleanest evidence is the **double lease read**. Within one evolve drain turn, `select` computes the live set via `_evolve_common.live_run_keys` (`evolve_select.py:214`), and `cli_main` recomputes it via `liveness_map` (`evolve_drain.py:140`). These two L snapshots can disagree, and the consequential decision — the marker-remove — keys on the *stale* select-time `live_runs` (`evolve_drain.py:137`), not the fresh `liveness_map` classify. Worse, the predicate itself differs by file: `_evolve_common.live_run_keys:121` keeps only `state == 'live'`, while `queue_drain._worktree_keys:94` deliberately keeps every run dir live-or-not (verified: same `feature-*/.flow/runs/*` pool, opposite filters). So "is key X in flight" has at least two answers depending on which join site and which predicate you ask. A single flock-consistent ledger read replaces all six unions with one snapshot, and there is exactly one definition of live.

**Worst TOCTOU windows, ranked by blast radius:**

1. **reap classify -> `gh pr merge` + `bd close` (§2f).** Irreversible and unguarded across the process boundary: it squash-merges onto main and closes the bead, and a run that acquires a lease in the classify->merge gap has its PR merged and bead closed out from under it. Only the *local* worktree teardown is lease-gated (classify_then); the merge and the close are not. Witnessed: flow-72d9 / flow-ztfv/#231.
2. **The session-cleanup central classify->mutate (§2e).** `claude stop` + `rm -rf` on a session that resumed in the gap. Bounded only because `claude stop` is resumable, but it got no `classify_then`-equivalent fix; it is the one destructive path with a fully lock-free read->act span and no under-flock re-check.
3. **The MASTER select union -> launch (§2f).** Double-launch or a second concurrent hot. Narrowed by the launch_ledger but structurally open because the marker is written after select, not atomic with it.
4. **The version stamp/remerge classify-then-push (§2c).** Duplicate version stamps that require a whole recovery script (`version_remerge`). Converges only because merges are prose-serial — a convention an operator can violate.

Pattern across all four: the destructive act runs in a *later, separate process* from the liveness read, and the only window that is actually closed is the one where the read and the mutation were forced under one flock (`classify_then`, `takeover_clear`). Every still-open window is one where they are not.

| **Select->launch double-launch + 2nd-hot races** | Closed structurally: a drain acquires the fleet flock, reads membership, decides the batch, and registers atomically; a second drain serializes behind it | The hot/anchor partition POLICY (<=1 hot, no shared primary anchor) and the `issue_type != epic` filter — the ledger gives a consistent snapshot to apply policy against, it does not make the policy unnecessary |
| **The §3 lease-heartbeat-during-CI-re-wait** | Folds into the ledger heartbeat | The mandatory post-push CI re-wait — only CI proves a textually-clean version merge is semantically correct; a ledger cannot |
| **Merge serialization (currently PROSE-ONLY)** | A flock-held merge-serialization token makes "process serially" a held invariant instead of a convention (child-4) | `version_remerge`'s git resolution logic (is this conflict version-only? what is next-from-main?) — intrinsic content arbitration, NOT coordination state. The ledger demotes version_remerge to a fallback, it does not remove it |
| **The classify-then-destructive-teardown windows** | — | `classify_then` / `takeover_clear` under-flock destructive teardown (worktree remove, quarantine) stays. Fleet-dereg does not make the worktree-remove safe; that needs the per-ticket lease flock held across classify+rm (flow-72d9). The ledger heartbeat is advisory; `run.lock` is authoritative for that worktree |
| **Work-state (R, B)** | — | `gh pr list` + `git for-each-ref` (R) and `bd` status (B) stay authoritative for PR/branch and bead lifecycle. The ledger covers run presence, not PR/bead terminal state. These are external systems the fleet flock cannot serialize |

**The one-line invariant.** The ledger subsumes "who is alive across the fleet" (the join). The per-ticket lease must stay as the mutex + crash-recovery + atomic-remediation primitive. Different granularity: the ledger answers "is a session working this key"; the lease answers "is THIS process the rightful owner of the state transitions."

## 5. AGENT-TEAMS BUILD-VS-ADOPT

The epic folds in an alternative: instead of building a flock ledger, adopt the harness's own coordination primitives (`TeamCreate` / `SendMessage` / `TaskList` / `Monitor`). The epic predicts this loses. It does, and the binding reason is an execution boundary, not a feature gap.

### The decisive constraint: where the liveness read executes

The current liveness consumers are **detached stdlib python subprocesses**. `evolve_drain.py` imports `evolve_select.select(...)` and calls `lease.classify()` directly on lock files; the result is a pure decision a script consumes. The harness agent primitives live in the **model's tool namespace**, not python's. A backgrounded engine script cannot invoke them. So "adopt the agent primitives" is not a like-for-like swap — it requires relocating the liveness decision out of the engine and up into the prose/agent layer. That detonates the `pure-core + thin-CLI` design the whole engine is built on (every coordination script: classify in python, run side effects in reviewable prose), and is a *larger* blast radius than the ledger it would replace. This single boundary decides most of the table.

### Option B is two different proposals

- **B1 = agent-tool coordination** (Teams / Monitor / TaskList as the authority).
- **B2 = the filesystem jobs registry** (`~/.claude/jobs/*/state.json`) promoted to the positive-registration authority. This is the honest steelman: already files, already positive register/dereg, already stdlib-readable.

### Pros / cons

| Axis | (A) Build `.flow/fleet/` flock ledger | (B1) Agent-Teams / Monitor / TaskList | (B2) `~/.claude/jobs` as authority |
|---|---|---|---|
| **Execution boundary** | Python module, called in-process by the drain subprocess — matches today's classify-in-script design | **Fails.** Agent-namespace tools; a detached engine script cannot call them. Forces liveness up into prose | OK — `glob` + `json.load` from python, same as today |
| **Stdlib-only runtime** | Pure stdlib (mirrors `lease.py`: flock + atomic write + json) | Interactively-authenticated harness surfaces, may be **absent in headless/cron** | Stdlib filesystem read |
| **Headless / cron survival** | Yes — flock + files have no TTY | **Witnessed failure** (below): harness fleet-introspection blocks on a TTY | Yes — files, no TTY |
| **Cross-reboot detached runs** | `boot_id` already in the lease model; ledger inherits reboot-clearability | **Fails structurally.** `TeamCreate` is session-scoped subagent coordination by a long-lived in-memory team-lead. Flow has N *top-level detached* sessions across **two** drain loops that must outlive reboots — no in-memory lead to coordinate them | Job dirs persist across reboot, but the schema carries **no `pid`/boot field** -> cannot distinguish a live run from a rebooted-away corpse without re-deriving from the lease anyway |
| **TOCTOU / serialization** | flock-serialized read->decide->write is the **existing safety idiom** (`run.lock.lock`, `machinery_edit.py`). Provable equivalence is achievable | No contract over the harness store; not serializable by flow | Schema-less, reverse-engineered, **no flock contract**, no atomic-write guarantee flow controls |
| **One-hot merge isolation** | Ledger can hold the merge-serialization token (child-4) | Would require the merge gate to live in the agent layer | No place to anchor a serialized token |
| **Net-new safety-critical code** | **Worst axis for A.** New machinery in exactly the category generating the bug class (6 coordination fixes on 2026-06-10) | Low new code, but unusable here | **Best axis.** Reuses a registration surface the platform already provides |
| **Schema stability** | Flow owns the schema | Flow owns nothing | **Flakiest source today**; child-5 exists to *retire* this surface |

### Primary-source evidence (the nail)

`evolve_session_cleanup.py` module docstring, verified verbatim on disk:

> "Enumeration + liveness are FILESYSTEM-ONLY. Never `claude agents --json`: it blocks on a TTY and the drain itself can run headless (flow memory, hit twice 2026-06-05). We scan `~/.claude/jobs/*/state.json` directly. [...] There is NO `pid` field, so no process-liveness check."

Flow **already tried** adopting a harness tool for fleet introspection. It failed the headless constraint, twice, and the team fell back to a raw filesystem scan. That is witnessed precedent against B1, and the "no `pid` field" note is exactly why B2 cannot carry liveness on its own — every consumer that reads the jobs dir *still* falls back to `lease.classify()` for the actual live/dead call.

### Blocking constraints that decide it

1. **Execution boundary** (detached python subprocess) eliminates B1 outright — the engine cannot call agent tools.
2. **Headless/cron + TTY-block**, twice-witnessed, independently eliminates B1.
3. **Cross-reboot detached fleet across two drain loops** has no in-memory coordinator, so the session-scoped Teams model structurally doesn't fit.
4. **Robustness floor** (CLAUDE.md: run lease, TOCTOU guard, atomic writes/quarantine are load-bearing, "simplify presentation, never the safety machinery"): B2's schema-less, flock-less, `pid`-less store cannot carry the TOCTOU and reboot-clearability properties; only a flow-owned flock store can.

### The recommendation, earned not assumed

A is the only option whose execution context, runtime, headless behavior, reboot semantics, and serialization contract all clear flow's hard constraints. B1 is unusable from the engine; B2 is the honest near-miss but cannot hold the load-bearing safety properties. The strongest adopt argument is real and should be recorded: B2 shows the platform already provides a positive register/dereg *surface*. Flow isn't adopting it as the **authority** (it can't — schema-less, no flock, no `pid`), but the fleet ledger should mirror its registration *shape* (per-key file, write-on-launch, remove-on-register/finish) rather than invent a novel one. That keeps the new machinery small and legible, which directly addresses A's only real weakness.

## 6. GO / NO-GO VERDICT + children 2-5 sizing

### Verdict: GO (staged), confidence ~0.78

The TOCTOU surface is broad (six independent join sites, ~13 distinct races, two of them irreversible and unguarded across process boundaries) and recurring (the epic's witnessed friction: six coordination fixes on 2026-06-10 alone, 36 of 246 commits churning the six coordination files, the same bug class reappearing in different paths — flow-ocp/#211 and flow-s85/#235 are the same takeover TOCTOU twice). That is broad and recurring enough to justify one unifying authority rather than continued point patches. And build beats adopt decisively on the constraints that matter (execution boundary, headless, reboot, provable serialization); the one axis adopt wins (least net-new code) is better captured by *imitating* the jobs-registry registration shape inside a flow-owned flock ledger than by *depending* on the harness store.

Why **go** and not **conditional**: nothing blocks child-2 from starting. The two conditions are design constraints the children must carry, not gates on the epic itself. The property-equivalence proof gates child-5 (retirement) only, and the staleness floor is a design constraint on child-3 — neither stops the build from beginning. Stated as the two non-negotiables:

1. **Keep a heartbeat-staleness fallback.** Positive deregistration alone blocks the drain forever on a crashed run (the epic's binding correction). Do not ship pure positive-dereg.
2. **Property-equivalence proof before any retirement.** The ledger may replace the lease/TOCTOU properties only once proven equivalent (CLAUDE.md robustness floor; the epic makes this binding for child-5 — proof, not just corpus green). Keep the `~/.claude/jobs` scan + `lease.classify` cross-check until the proof lands.

Confidence anchored to the epic's own 0.75; this spike confirms rather than moves it (the maps verified against source, the agent-teams precedent confirmed verbatim), so ~0.78. Not higher: A is net-new safety-critical machinery in exactly the category that has been generating the bug class, so execution risk is real and the staged sequencing + the child-5 proof are what keep it bounded.

### Children 2-5 sizing (evidence-driven)

- **Child-2 (fleet-ledger): stays as core, build first.** The `.flow/fleet/` flock-serialized register+heartbeat+dereg, subsuming launch markers. This is the foundation children 3-5 build on. Justified unchanged. Constraint to carry: mirror the jobs-registry registration *shape* (per-key file, write-on-launch, remove-on-register/finish), and keep the ledger lock on durable storage OUTSIDE any worktree (the version_remerge/lease maps note this also fixes the doomed-inode window, but only if the ledger lock is not torn down with the worktree).

- **Child-3 (readers-cutover): stays, conditioned on the staleness floor.** Selectors/reap/cleanup read the ledger. The condition is binding: reap requires positive dereg WITH a heartbeat-staleness fallback, else a crashed run blocks the drain forever. Additional constraint from the maps: the cutover must move the destructive act (`gh pr merge`/`bd close` for reap, `claude stop`/`rm -rf` for cleanup) to re-check the ledger UNDER the fleet flock right before the irreversible step — otherwise the rebuild has merely swapped one lock-free signal for another and the worst window (§3 #1) survives.

- **Child-4 (merge-token): stays, scope clarified to token-only.** A merge-serialization token in the ledger, making the currently prose-only "process serially" a held invariant. Clarify explicitly: this does NOT subsume `version_remerge`'s git-resolution logic (conflict-is-version-only detection, next-from-main computation) — that is intrinsic content arbitration and must stay. version_remerge is *demoted to a fallback*, not removed. The token must wrap the fetch+merge+push span, not just the gh-merge step, or main can still move between the version read and the push.

- **Child-5 (retirement): RE-SCOPE.** The epic preview says "retire superseded machinery (TTL blind-window logic, jobs-dir scan)." The session_cleanup map flags a regression that forbids full retirement of the jobs-dir scan: a run that crashes BEFORE its first ledger register is invisible to a pure ledger but visible today as a `~/.claude/jobs/<id>/` dir. So child-5 retires the TTL blind-window logic (`launch_ledger.py`) fully, but the jobs-dir scan must be DEMOTED to a reconcile/orphan-detection fallback, not deleted. And retirement is gated on the property-equivalence proof, not corpus green alone. Re-scope the bead title/description to "retire launch-ledger TTL logic; demote jobs-dir scan to reconcile fallback; gated on property-equivalence proof."
